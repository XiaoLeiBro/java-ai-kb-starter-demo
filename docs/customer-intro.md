# Java 企业 AI 知识库助手 — 客户介绍稿

> 用于给潜在客户、外包合作方、中小企业老板、技术负责人介绍本项目。内容偏业务表达，不强调代码细节。

---

## 一句话介绍

**Java 企业 AI 知识库助手** 是一个面向中小企业和传统 Java 系统的私有化知识库问答 Demo，可以把企业内部文档、制度、产品资料、客服话术等内容整理成可检索、可问答、可返回来源引用的 AI 助手。

---

## 它解决什么问题

很多中小企业都有类似问题：

```text
制度文档分散，员工不知道去哪里找
产品资料很多，新员工学习成本高
客服话术不统一，回答质量不稳定
历史项目资料沉淀在 Word / PDF / Markdown / TXT 中，无法快速复用
老板或主管经常被重复问题打扰
传统系统里有数据，但缺少自然语言查询入口
```

本项目提供一种轻量方案：

```text
上传资料 → 文档切分 → 向量化入库 → 自然语言提问 → 返回答案和引用来源
```

## 适合哪些企业

比较适合：

| 企业类型 | 典型场景 |
|---|---|
| 小型软件公司 | 项目文档、接口文档、交付文档问答 |
| 电商团队 | 商品资料、客服话术、售后规则问答 |
| 培训机构 | 课程资料、学员 FAQ、内部知识库 |
| 财税 / 咨询公司 | 政策资料、服务流程、客户常见问题 |
| 传统企业 IT 部门 | 内部制度、运维文档、系统说明问答 |
| 外包团队 | 作为 AI 知识库项目的二开模板 |

不适合：

```text
要求开箱即用的完整 SaaS 平台
没有任何文档沉淀的企业
希望 AI 直接替代专业人员做最终决策的场景
对权限、审计、数据合规有极高要求但又不愿做定制开发的场景
```

## 为什么选择 Java 技术栈

很多企业内部系统仍然是 Java / Spring Boot 技术栈。

相比重新引入一套 Python 服务，本项目更适合以下场景：

```text
已有 Java 后端团队
已有 Spring Boot / Spring Cloud 系统
希望 AI 能力接入现有业务系统
希望统一鉴权、日志、数据库、部署和运维体系
希望后续能二开成企业内部工具
```

项目采用：

```text
Spring Boot
PostgreSQL
pgvector
MyBatis-Plus
Docker Compose
DDD 分层结构
OpenAI Compatible 模型接口
```

## 和 Dify / FastGPT / MaxKB 的区别

成熟平台更适合快速搭建通用知识库。

本项目更适合：

```text
Java 后端团队学习 AI 工程化
企业已有 Java 系统的二次开发
私有化部署前的技术验证
外包项目交付前的工程模板
把 RAG 能力嵌入已有业务系统
```

它不是要替代成熟平台，而是回答：

> 如果企业原本就是 Java 技术栈，如何把 AI 知识库能力做进自己的系统里？

## 可演示流程

演示时可以按下面流程讲：

```text
1. 注册并登录系统
2. 创建一个知识库，例如“公司制度知识库”
3. 上传一份公司年假制度或客服售后规则文档
4. 系统自动完成解析、切分、向量化和入库
5. 在问答页面输入问题：公司的年假规则是什么？
6. 系统返回答案，并展示引用来源
7. 查看会话历史和调用记录
```

## 推荐交付方式

### 方式一：技术验证 Demo

适合客户先判断方向是否可行。

```text
部署一个可运行 Demo
导入少量客户文档
演示自然语言问答和引用来源
输出后续改造建议
```

### 方式二：小型私有化部署

适合小团队内部使用。

```text
部署到客户服务器
配置模型供应商 Key
接入企业文档
提供基础账号体系
提供使用说明
```

### 方式三：业务系统二开

适合已有 Java 系统的企业。

```text
接入客户现有用户体系
接入现有业务数据
定制权限过滤
定制问答场景
接入审计和运维体系
```

## 对外介绍话术

可以这样介绍：

```text
我做了一个 Java 企业 AI 知识库工程模板，主要面向已有 Java 技术栈的中小企业或外包团队。

它不是简单调大模型 API，而是跑通了从用户登录、知识库创建、文档上传、文本切分、向量化、pgvector 检索、Prompt 组装、AI 问答到引用来源返回的完整链路。

如果企业有大量制度文档、产品资料、客服话术、项目文档，可以先用这个 Demo 验证内部知识库问答是否有价值。后续可以按企业权限、文档格式、模型供应商和部署环境做定制。
```
