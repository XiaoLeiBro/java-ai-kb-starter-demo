# 常见问题

## 项目相关

**Q：这个项目可以直接拿去做公司项目吗？**  
A：不可以。免费版采用 Source Available License，禁止商业使用、客户交付、二次销售或改名发布。  
如需商用，请联系作者获取商业授权或企业版本，详见 [paid-version.md](paid-version.md)。

---

**Q：支持国内哪些大模型？**  
A：当前 `v0.2` 已通过 OpenAI Compatible 协议接入 Chat Model 和 Embedding Model。  
免费版只读取配置中的一组 Chat / Embedding 模型，不提供多供应商管理后台。你可以按账号情况配置兼容 OpenAI 接口的模型服务，例如 SiliconFlow、DeepSeek、Qwen 或本地 OpenAI Compatible 服务。

商业版会提供更多模型供应商适配、多模型配置、调用审计和成本统计。

---

**Q：为什么不用 Python？**  
A：这个项目的定位是**给 Java 程序员的 AI 工程化模板**。Java 生态和 Spring 技术栈在企业系统中有天然优势：

```text
权限体系
业务系统集成
事务一致性
工程规范
部署治理
JVM 监控体系
团队接受度
```

这个项目关注的是：

> Java 程序员如何把 AI 能力接入真实业务系统。

---

**Q：为什么第一版不做 Agent？**  
A：Agent 的调试成本、Token 消耗和不确定性都高于普通 RAG。多数企业知识库、AI 客服和内部助手的第一阶段需求，本质上还是 RAG。  
所以本项目会先把 RAG 主流程做扎实，再考虑 Agent 类能力。

---

**Q：免费版为什么不做完整管理后台？**  
A：管理后台、权限体系、成本报表、调用审计、部署脚本和二开文档，是从 Demo 走向真实交付的关键部分。  
免费版会逐步保持主流程可运行，商业版本会补齐完整交付能力。

---

## 技术相关

**Q：JDK 要求？**  
A：JDK 21 LTS。项目使用 Java 21 作为基础版本，方便与 Spring Boot 4.x 和现代 Java 语法保持一致。

---

**Q：为什么用 Spring Boot 4.x，而不是降到 3.x？**  
A：当前仓库是新项目，不是老系统迁移，主线使用 Spring Boot 4.x 更适合作为面向未来的 Java AI 工程化模板。  

但企业交付要看客户环境：如果客户现有系统、内部脚手架、网关、监控或私有组件仍停留在 Spring Boot 3.x，商业版可以提供 Spring Boot 3.x 兼容分支或迁移说明，默认以 3.5.x 为基线。

本项目的版本策略是：

```text
免费 Demo 主线：Spring Boot 4.x
商业源码 / 企业交付：可按客户环境提供 Spring Boot 3.x 兼容方案，默认以 3.5.x 为基线
不为了未知客户提前把主线降级
```

---

**Q：支持 MySQL 吗？**  
A：免费版默认使用 PostgreSQL + pgvector，因为它能把业务数据和向量数据放在同一个数据库实例里，降低本地部署成本。  
商业版可根据场景提供 MySQL + Milvus、PostgreSQL + pgvector 或其他向量检索方案。

---

**Q：Demo 的向量维度是多少？**  
A：默认是 `1024`，按 `BAAI/bge-m3` 的向量维度设计。Flyway V2 中的 pgvector 字段也是 `vector(1024)`。  
如果你使用其他 Embedding 模型，需要同步修改：

```text
向量表字段维度，例如 vector(768) 或 vector(1536)
application.yml 中的 ai-kb.vector.dimension
Embedding 模型名称
```

---

**Q：可以离线部署吗？**  
A：注册、登录、知识库元数据和本地文件存储可以离线运行。RAG 问答主流程默认需要调用你配置的大模型 / Embedding 服务。  
如果需要完全离线，需要把 OpenAI Compatible `base-url` 指向本地 Ollama、vLLM 或其他私有化模型服务，并保证 Chat 与 Embedding 都可用。企业版可以提供完整私有化部署方案。

---

**Q：为什么选 PostgreSQL + pgvector？**  
A：Demo 阶段优先考虑可运行性和部署成本。  
如果一开始就引入独立向量数据库，会增加部署复杂度。PostgreSQL + pgvector 更适合个人学习、Demo 演示和小规模验证。

---

**Q：为什么用 MyBatis-Plus？**  
A：MyBatis-Plus 在国内 Java 项目中接受度较高，适合面向国内 Java 程序员的工程模板。  
同时，项目会通过 PO、Assembler 和 Repository 隔离基础设施层，避免 MyBatis-Plus 注解侵入领域对象。

---

**Q：用户表需要一开始就分库分表吗？需要用 MyBatis-Plus 雪花算法吗？**  
A：当前免费 Demo 不需要。用户认证和最小 RAG 主流程都以单库 PostgreSQL 为默认方案，分库分表会让本地运行、教学理解和二开门槛都变高。  

如果商业版遇到真实大用户量、多租户或企业级水平扩展需求，可以引入雪花 ID 和分库分表。但建议通过领域 / 应用层的 `IdGenerator` 端口生成 ID，再由 MyBatis-Plus 原样保存，不建议让 PO 层使用 `ASSIGN_ID` 直接生成聚合根 ID。

---

**Q：知识库里的企业数据安全吗？**  
A：Demo 版数据存储在你本地的 PostgreSQL 中。  
但如果你配置了公网大模型 API，提问内容和检索片段可能会发送给模型服务商。  
如需完全私有化，包括不调用公网 LLM，请使用本地模型或企业版私有化方案。
